# 写作与文档工程工作流

覆盖论文撰写期与定稿期的全部技术工作流。按需读对应小节即可。

## 1. 开写之前:建术语对照表

英文论文/外文资料 → 中文毕业论文的场景,第一步是抽取全部专业术语,和学生确认一张"英文 ↔ 中文 ↔ 缩写"对照表(例:Mobile Edge Computing ↔ 移动边缘计算 ↔ MEC;Signal-to-Noise Ratio ↔ 信噪比 ↔ SNR)。此后论文、翻译、PPT、指导记录全部从这张表取词。术语漂移(同一概念两种译法)是盲审观感的硬伤,而事后统一的成本远高于事前。

## 2. 章节生成工作流(标准循环)

每章按这个循环走,对应 SKILL.md 三条铁律:

1. **读源**:打开英文原文 PDF 对应小节 + 相关代码文件,列出本章要覆盖的小节、公式清单(含编号)、图表清单;
2. **给映射**:向学生展示"论文小节 ↔ 原文小节 ↔ 代码/图文件"对应表,确认范围;
3. **写作**:中文学术语体(format-spec.md 第 7 节);英文原文是素材不是模板——按中文论文的章法重组,补全推导中省略的步骤(本科论文比会议论文更要求自包含),背景与动机部分要比原文展开;
4. **生成 docx**:require `scripts/njupt_docx_template.js`,公式位置放 LaTeX 源码段(见下节);
5. **校验渲染**:`validate.py` → `soffice.py` 转 PDF → `pdftoppm` 出图逐页看;
6. **交付确认**:第一章交付后等学生确认风格,再批量推进;每章附"待核对项"清单(没把握的公式、需要学生补充的实验数据)。

## 3. 公式工作流:LaTeX → MathType(本技能最重要的单项)

学生的最终公式由 Word + MathType 管理。Claude 的职责是给出**可一键转换的 LaTeX 源码**,严格对照英文原文转写。

**交付格式**:公式所在位置放一个独立段落,内容是纯 LaTeX 源码,Consolas(或 Courier New)等宽字体,可设浅灰底色以示区分;段落后(或同段右侧制表位)放编号"式(2-5)"。学生操作:选中代码 → MathType 菜单 "Toggle TeX"(或粘入 MathType 编辑器)→ 公式生成。

**LaTeX 写法规范**(保证 MathType 解析成功率):

- 用基础 LaTeX 命令:`\frac` `\sum` `\prod` `\int` `\sqrt` `\left( \right)` `\mathbf` `\boldsymbol` `\mathcal` `\mathbb`(MathType 新版支持)`\hat` `\tilde` `\bar` `\exp` `\log` `\max` `\min` `_{}` `^{}`;
- **不要用**:`align`/`equation` 环境(只给公式体)、`\begin{cases}`(改用 `\left\{ \begin{array}{ll} ... \end{array} \right.`)、自定义宏、`\text{}` 里的中文(说明文字放正文,不进公式)、`\nonumber`、`\label`;
- 矩阵用 `\begin{bmatrix} ... \end{bmatrix}` 或 array;
- 多行长公式拆成多条独立公式分别编号,比一条带对齐环境的巨型公式稳得多;
- 上下标完整加花括号:`p_{\ell,s}^{(E)}(t)` 而非 `p_\ell,s^(E)`;
- 行内变量(正文叙述中出现的 γ、**w**_k 等)同样给行内 LaTeX 微代码,标注"行内公式",由学生逐个转换——不要用 Unicode 数学字符(γₖ、ℂ、‖·‖)冒充,那些字体不一致且无法被 MathType 接管。

**严谨性要求**:每条公式与英文原文逐符号核对(下标、转置/共轭转置 H、期望/范数记号、求和范围);维度自洽(矩阵乘法两侧维度对得上);全文符号唯一(同一希腊字母不得在不同章表示不同量——这是定稿检查实际抓出过的问题)。**任何一条没有十足把握的公式,在文档中用灰色斜体注明"式(x-y) 请对照原稿核对"**,宁可多标不可漏标。

## 4. 参考文献核实流程

1. 从论文/任务书提取现有条目,确定既有风格基准(大小写、et al.、标点);
2. 新增每条先 web 搜索核实:标题、作者、刊名、年、卷(期)、页码、DOI 全对上才收;预印本(arXiv)能换正式版就换;
3. 按 GB/T 7714-2015 排版(样例见 format-spec.md 第 6 节),与基准风格逐项对齐;
4. 给出每条的**建议插入位置**(哪一章哪一节哪句话后加 `[n]`),编号按文内首现顺序重排;
5. 交付前查重复(同一文献以不同写法出现两次是定稿检查实际抓出过的问题)。

## 5. 回应导师批注(逐条工作流)

1. 提取批注:unpack docx 后读 `word/comments.xml`,或学生发批注截图;给批注编号①②③…;
2. 分类:文字类(扩写/删改/术语)、格式类(缩进/空白/编号)、图片类(合并/重画)、结构类(章节调整)、文献类(数量/质量);
3. 逐条给方案;**有两种以上改法时列出方案 A/B 及工作量对比,标注推荐项**。典型案例:
   - "多张独立图排版松散,建议合并为组合图" → 方案 A(推荐):MATLAB `tiledlayout` 合并为一张带 (a)(b)(c) 的组合图,加总图题,正文几乎不改;方案 B:拆为独立图重新编号——会引发全文图号连锁改动,不推荐;
   - "本章小结过于简略,需扩充" → 扩为 2~3 段以上,每个实验维度一段"结果 → 结论 → 意义",末段总括;
   - "参考文献不够 30 篇" → 按论文主题方向分组补充(每个技术线 2~3 篇),走第 4 节核实流程;
4. 改完输出"批注 ↔ 修改位置 ↔ 改法"对照表,方便学生向导师复命。

## 6. 查重(降重复率)

- 第一道防线是引用规范:综述性段落改写转述+标引用,不整段搬运(哪怕是自己发表过的论文,本科论文查重同样会标);
- 被标红段落的改写:换句式骨架(主动↔被动、拆并句)、换叙述顺序、术语保留但连接词与论证路径重写;
- 公式、图表、参考文献列表通常不计或单独处理,精力集中在正文连续文字;
- 改后语义必须不变——降重不是同义词替换游戏,改完读一遍是否还是原意、是否仍通顺。

## 7. AIGC 检测(降 AI 率)

近年新增检查,检测报告会把疑似 AI 生成段落标色分级(实测案例:红色 `F12828` 高风险、橙色 `F39800` 中风险——具体色值因检测系统而异,以报告为准)。

**提取被标段落**:学生上传带标色的检测版 docx → unpack → 在 `word/document.xml` 里按高亮/底纹颜色值检索 `w:highlight` 或 `w:shd w:fill`,把命中段落原文连同位置抽出来,形成"待改写清单"。

**改写策略**(对每个被标段落):

- 打破对称排比:"首先…其次…最后…""一方面…另一方面…"这类整齐结构是高权重特征,改成不等长的自然推进;
- 替换 AI 高频套话:如"应运而生""奠定了基础""随着…的不断发展""值得注意的是""综上所述"(章末小结除外)等,换成具体表述;
- 长短句交替:连续的长复合句拆短,偶尔用短句收束;
- 名词化 → 动词化:"对…进行了优化"→"优化了…";"…的实现依赖于…"→"…靠…实现";
- 降低"本文"密度:同一段反复"本文…本文…"改用"所提算法""该方案"或直接省略主语;
- 加入具体性:笼统的"性能显著提升"换成图表里的真实量级与条件。

**红线**:专业术语、公式与编号、图表编号、引用标注 `[n]`、数据数值**一律原样保留**。还有一条容易忽视的红线——降 AI 率常把笼统句改具体以削弱"AI 味",但**只能用源材料(图、表、原文)里真实存在的结论和数值来具体化**,绝不能为了让句子更实在而编造一个"看起来合理"的提升幅度或对比结论;若原文该处本就只有定性表述、学生也没提供对应数据,就保持定性,换句式而非换内容。改写按章交付,第一章过学生确认后再继续。改完建议学生重测一次再定稿。

## 8. 目录(TOC)故障修复

症状:"只更新页码"后页码错乱,"更新整个目录"后自定义格式全部丢失。

根因:① 正文标题没有套 Heading 样式(只是手动加粗的普通段落),Word 收集不到正确条目;② 目录的字体格式直接刷在文字上而没存进 TOC 1/2/3 段落样式,重建目录时被默认样式覆盖。

修复路径(按代价从小到大):

1. **治本(还在写作期)**:所有标题套 Heading 1/2/3(docx-js 里 heading + `outlineLevel`),目录格式改到 TOC 样式里 → 以后随便更新;
2. **直接修目录(临近交稿,学生不愿大动)**:unpack → 在 document.xml 中定位 TOC 域,核对每个条目超链接锚点(`w:anchor`/书签)与正文书签是否对应、`PAGEREF` 页码是否与渲染 PDF 的真实页码一致,直接改 XML 中的页码文本 → pack → 渲染核对。**只动目录,正文一个字不碰**;
3. 提醒学生:XML 直改的页码是静态的,之后正文若增删页需重修。

## 9. 仿真图规范(MATLAB,论文级)

把样式集中到一个 `build_paper_style()` 函数统一管理(实测验收过的参数):

```matlab
style.font_name = 'Times New Roman';
style.axes_font_size  = 12;   % 坐标轴刻度
style.label_font_size = 13;   % 轴标签
style.title_font_size = 14;   % 图内标题(论文图可省,标题放Word图题)
style.legend_font_size = 11;
style.line_width = 2.0;  style.highlight_line_width = 2.8;  % 主方案曲线加粗
style.marker_size = 5.2; style.grid_alpha = 0.15;
% 配色:每个方案固定一种颜色,全篇所有图保持一致,如
% 所提方案深红 [0.72 0.20 0.22],基准蓝 [0.29 0.56 0.89] 等
```

要点:

- 同一方案在全部图中用**同一颜色+同一线型+同一图例名**(图例名与正文方案名一字不差,如 "Proposed Algorithm A");
- 曲线加 marker 并设 `MarkerIndices` 抽稀,避免 marker 糊成一团;`box on; grid on`;
- 导出双份:`.fig`(便于以后只改样式不重跑仿真——写一个 restyle 脚本批量读 .fig 改样式再导出,比重跑仿真便宜得多)+ `.png`(`exportgraphics(gcf,'x.png','Resolution',300)`);
- 图内文字一律英文(Times New Roman),中文图题在 Word 里加;
- 多子图合并用 `tiledlayout(3,1,'TileSpacing','compact')`,子图左上角标 (a)(b)(c);
- **图的解读必须来自真实图片**:写正文分析前先 view 实际 png,描述真实的拐点位置、收敛值、相对排序;两份"看起来差不多"的图,数值细节可能完全不同。

## 10. Python/JS 文档生成踩坑速记

- docx-js:中文内容字符串**一律用反引号模板字符串**(内容里的英文双引号会炸普通字符串);`TextRun` 同时设 `font: { name: "Times New Roman", eastAsia: "宋体" }`;PageBreak 必须包在 Paragraph 里;
- python-docx 同理:`run.font.name = 'Times New Roman'` 之外还要 `run._element.rPr.rFonts.set(qn('w:eastAsia'), '宋体')`;
- 从 docx 提取文本时 MathType 公式(OLE 对象)会整体丢失——所以**公式永远以英文原文 PDF 为准源**,不要从学生的旧 docx 文本里恢复公式;
- 渲染核对命令组:`soffice.py --headless --convert-to pdf x.docx && pdftoppm -jpeg -r 110 x.pdf page`,然后逐页 view。
